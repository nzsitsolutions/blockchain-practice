¿Qué es un smart contract?

Un smart contract es código que vive en la blockchain. Una vez deployado, se ejecuta automáticamente cuando alguien lo llama, sin intermediarios, sin posibilidad de que nadie lo detenga o modifique (a menos que el propio código lo permita).

La analogía de la máquina expendedora

Un smart contract es exactamente como una máquina expendedora:

Máquina expendedora
Metés $1.50
Elegís producto B3
La máquina ejecuta la lógica
Te da el producto
Sin empleado, sin negociación
Si no tenés $1.50, no sale nada
Smart contract
Mandás €1000 + firma
Llamás a buyTokens()
El contrato ejecuta la lógica
Te da 1000 tokens
Sin intermediario, sin negociación
Si no tenés el balance, revierte
La diferencia clave con una API: tu API de .NET la podés apagar, modificar, hackear o engañar. Un smart contract deployado en Ethereum no lo puede apagar nadie. Ni vos. Ejecuta exactamente lo que dice el código, siempre, para siempre. Por eso el código tiene que estar perfecto antes del deploy.
Comparativa: tu API .NET vs un smart contract
API .NET Smart Contract
¿Dónde vive? Tu servidor / AWS En miles de nodos de la blockchain
¿Quién lo ejecuta? Tu servidor Todos los nodos simultáneamente
¿Se puede modificar? Sí, deployás nueva versión No (salvo patrón upgradeable)
¿Se puede apagar? Sí No (salvo función pause())
¿Quién paga la ejecución? Vos (AWS bill) El que llama la función (gas)
¿Es auditable? Solo si vos lo permitís Sí, cualquiera ve el código
¿Qué pasa si hay un bug? Deployás hotfix El bug vive para siempre (o pausás)
El bug más caro de la historia: en 2016, un bug en el smart contract "The DAO" permitió a un atacante drenar $60 millones en ETH. El contrato ejecutó exactamente lo que decía el código — el problema era que el código tenía un error. No había hotfix posible. Esto llevó al hard fork de Ethereum y la creación de Ethereum Classic.

EVM — Ethereum Virtual Machine

La EVM es el "motor" que ejecuta los smart contracts. Es una computadora virtual que corre en cada nodo de Ethereum simultáneamente, garantizando que todos lleguen exactamente al mismo resultado.

La analogía de la calculadora distribuida

Imaginá que tenés 1000 calculadoras idénticas en distintos países. Cada una recibe exactamente los mismos números y operaciones. Si todas llegan al mismo resultado, ese resultado es verdad. Si alguna da un resultado distinto, está rota o mintiendo y la red la ignora.

La EVM es esa calculadora, corriendo en miles de nodos simultáneamente. Tu smart contract es la operación que le mandás a todas a la vez.

Por qué esto importa: cuando tu contrato ejecuta buyTokens(), esa función se ejecuta en miles de nodos al mismo tiempo. Todos llegan exactamente al mismo estado final. Eso es lo que hace que el resultado sea confiable sin necesitar un servidor central.
De Solidity a bytecode — el flujo de compilación

Vos (o el especialista) escribe en Solidity. Pero la EVM no entiende Solidity. El compilador lo traduce a bytecode, que es lo que realmente se ejecuta.

// Lo que escribe el especialista (Solidity)
function buyTokens(uint256 amount) public {
  require(whitelist[msg.sender], "No KYC");
  balances[msg.sender] += amount;
}

// Lo que ejecuta la EVM (bytecode)
60 80 60 40 52 34 80 15 61 00 10 57 60 00 80 fd...
// (ilegible para humanos, perfecto para máquinas)
Para vos como tech lead: nunca vas a leer bytecode. Vas a leer Solidity. Pero es importante saber que lo que se deploya en la blockchain es bytecode, y que si el código fuente Solidity no está verificado públicamente, nadie puede auditar qué hace el contrato realmente.
Estado global — la "base de datos" de la EVM

La EVM mantiene un estado global: un registro de todos los balances, todas las variables de todos los contratos, en todos los nodos. Cuando una transacción ejecuta una función, modifica ese estado.

// El estado que guarda tu contrato en la blockchain
// (simplificado)

mapping(address => uint256) public balances;
// balances[0xABC] = 1000 ← Juan tiene 1000 tokens
// balances[0xDEF] = 500 ← María tiene 500 tokens

mapping(address => bool) public whitelist;
// whitelist[0xABC] = true ← Juan tiene KYC
// whitelist[0xGHI] = false ← Carlos no tiene KYC

bool public paused = false;
// El contrato está activo

Este estado vive en miles de nodos simultáneamente. Es la "base de datos" de tu plataforma que nadie puede alterar unilateralmente.

Gas — el combustible de la EVM

Cada operación que ejecuta la EVM tiene un costo en gas. Sumar dos números: 3 gas. Guardar un valor en storage: 20.000 gas. Deployar un contrato: 1-3 millones de gas.

Operación Gas aprox. Por qué cuesta eso
Suma/resta 3 Operación simple en CPU
Leer del storage 800 Todos los nodos deben leer
Escribir en storage 20.000 Todos los nodos deben guardar para siempre
Transferir tokens (ERC-20) ~65.000 Leer + escribir + emitir evento
Deploy de contrato 1-3M Guardar todo el bytecode en la red
Para tu proyecto: escribir en storage es lo más caro. Por eso vas a guardar solo lo estrictamente necesario on-chain (balances, whitelist, estado del contrato). Todo lo demás (datos del inmueble, fotos, documentos) va off-chain en tu BD o en IPFS.

Eventos — el sistema de logs de blockchain

Los eventos son registros inmutables que el contrato emite cuando algo importante sucede. Tu backend los escucha para saber qué pasó on-chain.

// Definición de eventos en el contrato
event TokensPurchased(
  address indexed buyer,
  uint256 amount,
  uint256 timestamp
);

event WalletFrozen(address indexed wallet);
event RentDistributed(uint256 totalAmount);

// Dentro de buyTokens(), el contrato emite:
emit TokensPurchased(msg.sender, amount, block.timestamp);
Para tu backend .NET: los eventos son tu WebSocket de la blockchain. Tu backend se suscribe a eventos específicos via Alchemy/Infura y reacciona cuando ocurren. Cuando el contrato emite TokensPurchased, tu backend lo recibe y actualiza el estado en Postgres.
// Tu backend C# escuchando eventos
var eventHandler = contract
  .GetEvent<TokensPurchasedEventDTO>()
  .CreateFilterInput();

var logs = await web3.Eth.Filters
  .GetLogs.SendRequestAsync(eventHandler);

foreach (var log in logs) {
  // Actualizar Postgres con la compra confirmada
  await UpdatePurchaseStatus(log.Buyer, log.Amount);
}
Modificadores — los guardias de seguridad del contrato

Un modificador es código que se ejecuta antes de la función. Es el middleware de Solidity. Si el modificador falla, la función no se ejecuta.

// Definición de modificadores
modifier onlyOwner() {
  require(msg.sender == owner, "Solo el admin");
  _; // acá se ejecuta la función
}

modifier onlyWhitelisted() {
  require(whitelist[msg.sender], "Sin KYC");
  _;
}

modifier whenNotPaused() {
  require(!paused, "Contrato pausado");
  _;
}

// Usados en funciones:
function buyTokens(uint256 amount)
  public
  onlyWhitelisted ← verifica KYC
  whenNotPaused ← verifica que no esté pausado
{
  // solo llega acá si pasó ambos modificadores
  balances[msg.sender] += amount;
}
La analogía: los modificadores son como los guardias en la entrada de un edificio. Antes de que entre cualquiera a la función, los guardias verifican: ¿tenés KYC? ¿el contrato está activo? ¿sos el admin? Si algún guardia dice no, te rechaza en la puerta y la función nunca se ejecuta.

Estándares ERC — los contratos que vas a usar

ERC significa "Ethereum Request for Comments". Son estándares acordados por la comunidad que definen cómo debe comportarse un tipo de contrato. Si tu contrato implementa un estándar, cualquier wallet, exchange o plataforma puede interactuar con él automáticamente.

Los tres estándares relevantes para tu proyecto
¿Cuál usás en Ladrillo?
ERC-20 ERC-1400 ERC-3643
Transfer restrictions No Sí Sí
KYC / Whitelist No nativo Parcial Sí, nativo
Freeze de wallet No Sí Sí
Compliance regulatorio No Básico Completo (MiCA)
Adoptado en España No para esto Algunos Reental, otros
Recomendación Ladrillo ❌ ⚠ posible ✓ ideal
Conclusión: ERC-3643 (implementado por Tokeny) es el estándar de facto para security tokens regulados en Europa. Incluye KYC on-chain, transfer restrictions, freeze, y está diseñado explícitamente para cumplir con MiCA. Es lo que usás en Ladrillo y lo que el especialista blockchain debería implementar.

Conectar tu backend .NET con los contratos

Acá es donde tu experiencia de backend entra en juego. Conectar un contrato a tu API .NET es más parecido a integrar una API externa que a programar blockchain.

La librería: Nethereum

Nethereum es la librería .NET equivalente a ethers.js. Te permite conectarte a la blockchain, leer estado de contratos, y enviar transacciones desde C#.

// NuGet: Install-Package Nethereum.Web3

var web3 = new Web3("https://polygon-rpc.com");
var contract = web3.Eth.GetContract(abi, contractAddress);

// Leer balance de un inversor (gratis, no gasta gas)
var balanceFunc = contract.GetFunction("balanceOf");
var balance = await balanceFunc
  .CallAsync<BigInteger>(investorAddress);

// Llamar función que escribe (gasta gas, requiere firma)
var whitelistFunc = contract.GetFunction("addToWhitelist");
var receipt = await whitelistFunc
  .SendTransactionAndWaitForReceiptAsync(
    adminWallet, gas, null, investorAddress
  );
Leer vs Escribir — la diferencia más importante
Leer (call) Escribir (send)
Ejemplos balanceOf(), isWhitelisted() buyTokens(), freeze(), transfer()
¿Gasta gas? No, gratis Sí, siempre
¿Necesita firma? No Sí, clave privada
¿Resultado inmediato? Sí No, esperar confirmaciones
¿Modifica el estado? No Sí
La regla de oro: si solo necesitás saber algo (¿cuántos tokens tiene Juan?), es una lectura gratuita e instantánea. Si necesitás cambiar algo (agregar a whitelist, transferir tokens), es una escritura que cuesta gas y necesita confirmaciones. Diseñá tu API sabiendo esto.
El flujo completo de una compra — tu backend orquesta todo
// 1. Inversor llama a tu API
// POST /api/tokens/buy { amount: 1000 }

// 2. Tu backend valida
var investor = await _db.Investors.FindAsync(userId);
if (investor.KycStatus != "APPROVED") return Forbid();

// 3. Privy firma la tx del inversor
var txHash = await _privy.SignAndSend(
  investor.WalletAddress,
  "buyTokens",
  amount: 1000
);

// 4. Guardás estado pendiente en Postgres
_db.Purchases.Add(new Purchase {
  TxHash = txHash,
  Status = "PENDING",
  Amount = 1000,
  InvestorId = investor.Id
});

// 5. Job escucha el evento TokensPurchased
// Cuando llega con 12 confirmaciones:
purchase.Status = "CONFIRMED";
await _hub.NotifyInvestor(investor.Id, "✓ Tokens acreditados");
ABI — el "contrato de interfaz" entre tu backend y el contrato

El ABI (Application Binary Interface) es un JSON que describe todas las funciones y eventos del contrato. Tu backend lo necesita para saber cómo encodear y decodear las llamadas.

// ABI simplificado del contrato de Ladrillo
[
  {
    "name": "buyTokens",
    "type": "function",
    "inputs": [{ "name": "amount", "type": "uint256" }],
    "stateMutability": "payable"
  },
  {
    "name": "balanceOf",
    "type": "function",
    "inputs": [{ "name": "account", "type": "address" }],
    "outputs": [{ "type": "uint256" }],
    "stateMutability": "view"
  }
]
En la práctica: el especialista te entrega el ABI junto con la dirección del contrato deployado. Con esos dos datos, tu backend puede interactuar con el contrato completamente. Es como recibir la documentación de una API externa.
